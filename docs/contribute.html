📋 Before You Start

1

Request membership of the Latent-Fields GitHub organisation to access the private repos.

↗ Request Latent-Fields membership
2

You need Python 3.10–3.12 with PyTorch 2.x, and Git. Python 3.13+ is not yet supported by PyTorch CUDA builds — install 3.12 if in doubt.

3

Clone both repositories into the same folder (they must be siblings — the runner syncs results to REE_assembly automatically):

git clone https://github.com/Latent-Fields/ree-v3.git git clone https://github.com/Latent-Fields/REE_assembly.git cd ree-v3 pip install torch numpy scipy --index-url https://download.pytorch.org/whl/cu124
⚗ Optional: automated setup script (experimental, not fully tested)
Once Python 3.12 and Git are installed (steps 1–2 above), you can run a setup script that handles the rest — cloning, pip install, git identity, CUDA check, and machine registration — in one guided session.

Download setup_contributor.py from the repo, then:
py -3.12 setup_contributor.py
The script will prompt for your details and print the runner start command at the end. If anything goes wrong, continue with the manual steps below.

⚙️ Start the Runner ✓ Low priority

1

From inside the ree-v3/ directory, start the runner at low CPU priority:

nice -n 10 /opt/local/bin/python3 experiment_runner.py --auto-sync --loop
nice -n 10 limits CPU scheduling priority. If your Python isn't at that path, use nice -n 10 python3 experiment_runner.py --auto-sync --loop
2

To keep it running after you close Terminal, use caffeinate + nohup:

nohup nice -n 10 /opt/local/bin/python3 experiment_runner.py --auto-sync --loop > runner.log 2>&1 &
3

To stop it:

pkill -f experiment_runner.py

1

Verify GPU is visible to PyTorch:

Must print True and your GPU name. If False, update NVIDIA drivers.

4

Stop:

Get-Process python | Where-Object {$_.MainWindowTitle -eq "" } | Stop-Process

🧪 What Happens First: Onboarding Verification

When your machine is registered, a short pipeline smoke test is queued specifically for your machine. It will be the first experiment your runner claims — before any scientific experiments.

→

What it does: runs 25 episodes of the REE training loop (20 warmup + 5 eval, 100 steps each — typically 3–8 minutes depending on your GPU). It verifies that imports work, the environment runs, the agent trains, and results are written and pushed correctly.

→

It also records your machine's throughput — GPU name, CUDA availability, and steps per second — in the result JSON. This feeds into smarter experiment scheduling over time.

→

PASS means the full round trip works: your machine ran an experiment, wrote the result, and pushed it to GitHub. You can confirm this by refreshing the project explorer — the result will appear under Experiments with your machine name. FAIL means something needs attention — check the runner terminal output for details.

The smoke test is assigned to your machine by name (machine_affinity) so no other machine will claim it. After it completes, the runner moves on to the shared scientific experiment queue automatically.

🔄 How Multi-Machine Coordination Works

→

The runner reads from a shared experiment queue on GitHub. Before starting each experiment, it writes a claim and pushes to GitHub — whichever machine pushes first runs the experiment. This prevents duplicate work without any central server.

→

Results are pushed back to GitHub automatically. The project maintainer reviews results and updates the claims registry.

→

Claims older than 6 hours are treated as stale (machine went offline) and become available for other machines.

Your contribution is tracked in contributions.json — compute hours, experiment count, and estimated cost (electricity + cloud-equivalent). This is updated each governance cycle.
