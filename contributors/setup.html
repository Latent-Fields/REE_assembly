⚙️ Start the Runner ✓ Low priority

What the runner flags do
  • --auto-sync — pull queue updates from GitHub before each pass; push results / claims back. Required for multi-machine coordination.
  • --loop — keep polling for new experiments after the queue empties. Without this the runner exits when there's nothing to do.
  • --remote-control — emit a per-tick heartbeat and accept commands (pause, stop, kick, release_claim) from the Machines dashboard. Default in the recommended command so your machine is visible and controllable from the explorer.
1

From inside the ree-v3/ directory, start the runner at low CPU priority:

nice -n 10 /opt/local/bin/python3 experiment_runner.py --auto-sync --loop --remote-control
nice -n 10 limits CPU scheduling priority. If your Python isn't at that path, use nice -n 10 python3 experiment_runner.py --auto-sync --loop --remote-control
2

To keep it running after you close Terminal, use caffeinate + nohup:

nohup nice -n 10 /opt/local/bin/python3 experiment_runner.py --auto-sync --loop --remote-control > runner.log 2>&1 &
3

To stop it:

pkill -f experiment_runner.py
1

Install Docker + NVIDIA Container Toolkit (required for GPU passthrough):

curl -fsSL https://get.docker.com | sh curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list sudo apt-get update && sudo apt-get install -y nvidia-container-toolkit sudo nvidia-ctk runtime configure --runtime=docker && sudo systemctl restart docker
2

Set a short stable name for this machine — used for experiment routing and contribution tracking:

export REE_MACHINE_NAME="my-machine-name"
Add to ~/.bashrc or ~/.zshrc to persist across sessions.
3

From the REE_assembly/docker/ directory, start the container:

docker compose up -d
-d runs it in the background. View logs with docker compose logs -f.
4

Open the Explorer to start the runner: http://localhost:8000/explorer

Bare-metal setup (no Docker)
1

Start at low priority from inside ree-v3/:

nice -n 10 python3 experiment_runner.py --auto-sync --loop --remote-control
2

Optional: run as a systemd user unit. Save as ~/.config/systemd/user/ree-runner.service:

[Unit] Description=REE Experiment Runner After=network.target [Service] WorkingDirectory=%h/ree-v3 ExecStart=/usr/bin/python3 experiment_runner.py --auto-sync --loop --remote-control Nice=10 Restart=on-failure RestartSec=30 [Install] WantedBy=default.target
Enable: systemctl --user enable --now ree-runner  |  Stop: systemctl --user stop ree-runner
1

Install Docker Desktop for Windows if you haven't already.

  • Enable WSL2 backend during install (recommended by Docker)
  • For GPU: update NVIDIA drivers to ≥527.41 — Docker Desktop handles the rest automatically
2

Open PowerShell and set a short stable name for this machine — used for experiment routing and contribution tracking. Replace my-machine-name with something like alice-rtx4090:

$env:REE_MACHINE_NAME = "my-machine-name"
Set this once per session, or add it to your PowerShell profile to make it permanent.
3

From the REE_assembly\docker\ directory, start the container:

docker compose up
The first run pulls the image and clones repos — takes a few minutes. Subsequent starts are instant.
4

Open the Explorer to start the runner:

http://localhost:8000/explorer
Click Start V3 Runner in the Experiments tab. Or set AUTO_START_RUNNER=true in docker-compose.yml to skip this step.
5

To stop:

docker compose down
Manual setup (bare-metal PowerShell, no Docker)
1

Verify GPU is visible to PyTorch:

python -c "import torch; print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0))"
Must print True and your GPU name. If False, update NVIDIA drivers.
2

Navigate to ree-v3\ and start at below-normal priority. Replace my-machine-name with a short stable ID:

$proc = Start-Process python -ArgumentList "experiment_runner.py --auto-sync --loop --remote-control --machine my-machine-name" -PassThru -NoNewWindow $proc.PriorityClass = [System.Diagnostics.ProcessPriorityClass]::BelowNormal
3

Background + log output:

$proc = Start-Process python ` -ArgumentList "experiment_runner.py --auto-sync --loop --remote-control --machine my-machine-name" ` -RedirectStandardOutput runner.log ` -RedirectStandardError runner_err.log ` -PassThru -WindowStyle Hidden $proc.PriorityClass = [System.Diagnostics.ProcessPriorityClass]::BelowNormal Write-Host "Runner started, PID $($proc.Id)"
4

Stop:

Get-Process python | Where-Object {$_.MainWindowTitle -eq "" } | Stop-Process

🔄 How Multi-Machine Coordination Works

→

The runner reads from a shared experiment queue on GitHub. Before starting each experiment, it writes a claim and pushes to GitHub — whichever machine pushes first runs the experiment. This prevents duplicate work without any central server.

→

Results are pushed back to GitHub automatically. The project maintainer reviews results and updates the claims registry.

→

Claims older than 6 hours are treated as stale (machine went offline) and become available for other machines.

Your contribution is tracked in contributors/contributions.json — compute hours, experiment count, and estimated cost (electricity + cloud-equivalent). This is updated each governance cycle.
